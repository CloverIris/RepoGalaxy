# RepoGalaxy Git 功能集成设计

> 状态：讨论中 | 最后更新：2026-03-13

---

## 1. 核心问题：RepoGalaxy 是什么类型的工具？

### 1.1 产品定位抉择

| 定位 | 描述 | 复杂度 | 竞品 |
|------|------|--------|------|
| **A. GitHub 浏览器** | 专注于发现、浏览、管理仓库 | ⭐⭐ | GitHub Desktop (轻量版) |
| **B. Git 客户端** | 完整的 Git 操作能力 | ⭐⭐⭐⭐⭐ | Sourcetree, Fork, GitKraken |
| **C. 混合体** | 浏览器 + 基础 Git 操作 | ⭐⭐⭐ | GitHub Desktop |

**建议定位：A + 轻量 C**

理由：
1. **核心竞争力**：发现算法 + 星空体验 ≠ Git 操作
2. **竞品差异化**：不做另一个 Git 客户端
3. **用户需求**：发现 → Clone → 用专业工具开发
4. **开发成本**：Git 操作库复杂，容易陷入细节

## 2. Git 功能范围 (建议)

### 2.1 第一层：集成 (必须)

```
与本地 Git 仓库的关联：

• 扫描本地仓库
  - 自动检测 ~/Code, ~/Projects 等目录
  - 识别 .git 文件夹
  - 关联到对应的 GitHub 仓库

• 显示仓库状态 (只读)
  - 当前分支
  - 是否有未提交更改
  - 最后提交时间
  - 与远程的差异 (ahead/behind)

• 快捷打开
  - 用 VS Code 打开
  - 用 JetBrains IDE 打开
  - 在终端中打开
```

### 2.2 第二层：轻量操作 (可选 Phase 2)

```
基础 Git 操作 (简化版)：

• Clone
  - 标准 Clone
  - Shallow Clone (只取最新)
  - 选择本地路径

• 快捷操作
  - Pull (拉取最新)
  - Fetch (获取更新)
  - 一键同步 (Fetch + Pull)

• 分支切换 (只读)
  - 查看本地分支列表
  - 切换分支 (如果工作区干净)
```

### 2.3 第三层：高级操作 (不建议)

```
复杂的 Git 操作 (暂时不做)：

✗ Commit (涉及暂存区、提交信息)
✗ Push (需要处理冲突)
✗ Merge/Rebase (太复杂)
✗ Stash (需要理解暂存概念)
✗ 历史查看 (GitHub 网页已很好)
✗ Blame/Log (专业工具更好)
```

## 4. 技术实现方案

### 4.1 Git 库选择

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| **LibGit2Sharp** | 功能完整，原生绑定 | 依赖 native 库，跨平台复杂 | ⭐⭐⭐ |
| **Process 调用** | 简单，使用系统 Git | 需要用户安装 Git | ⭐⭐⭐⭐ |
| **NGit** | 纯 C#，跨平台 | 功能有限，更新慢 | ⭐⭐ |

**建议：Process 调用 (系统 Git)**

理由：
- 开发者通常已安装 Git
- 无需处理 native 依赖
- 命令行输出稳定
- 跨平台简单 (统一命令)

### 4.2 只读状态检测命令

```csharp
// 获取当前分支
git rev-parse --abbrev-ref HEAD

// 检查工作区状态
git status --porcelain

// 获取与远程的差异
git rev-list --left-right --count HEAD...@{upstream}

// 获取最后提交信息
git log -1 --format="%h|%s|%cr"
```

### 4.3 轻量操作命令

```csharp
// Clone
git clone --depth 1 <url> <path>  // Shallow clone

// Pull
git pull

// Fetch
git fetch

// 切换分支
git checkout <branch>
```

## 5. 与 GitHub 的关联

### 5.1 自动关联逻辑

```
扫描本地仓库时:

1. 读取 .git/config 中的 remote.origin.url
2. 解析出 owner/repo
3. 查询 GitHub API 获取仓库信息
4. 本地显示时合并 GitHub 数据 (stars, description 等)
5. 建立 local_path <-> github_repo 映射
```

### 5.2 双向导航

```
GitHub 页面 → 本地：
• 如果本地已 Clone，显示"在 VS Code 打开"
• 如果未 Clone，显示"Clone 到本地"

本地页面 → GitHub：
• 始终提供"在 GitHub 查看"按钮
• 点击跳转对应仓库/分支
```

## 9. 结论

**建议的 Git 集成策略**：

> **"轻量级关联，不做重型客户端"**

- ✅ 扫描和展示本地仓库
- ✅ 显示只读的 Git 状态
- ✅ Clone、Pull、Fetch 等轻量操作
- ✅ 快速打开到专业 IDE
- ❌ 复杂的 Commit、Merge、冲突解决
- ❌ 完整的 Git 历史浏览

这样 RepoGalaxy 专注于「发现 → 获取 → 交给专业工具」的闭环，而不是成为另一个 Git 客户端。
